# SC-PROMPT-0001 Topological Memory Runtime Adoption v0

- **prompt_id:** SC-PROMPT-0001
- **status:** active
- **task_type:** integration + benchmark + interface hardening
- **surface_class:** experimental
- **authority_class:** advisory
- **prompt_class:** execution-prompt
- **claim_tier_ceiling:** defensible

## 1) Objective

Move `topological-memory-v0` one step along a disciplined engineering gradient:

from
- high-quality continuity enclave

to
- bounded, inspectable subsystem with at least one standard host workflow consumer.

This packet does **not** ask the implementer to prove grand Sandy Chaos theory.
It asks for a narrow systems result:

> make topological retrieval usable in one real repo-native workflow without granting it unauthorized runtime authority.

## 2) Source surfaces (`source_surfaces`)

Primary:
- `docs/17_endosymbiosis_and_host_assimilation.md`
- `docs/19_surface_authority_architecture.md`
- `spine/subsystems/SC-SUBSYSTEM-0001-topological-memory-v0.yaml`

Secondary:
- `docs/07_agentic_automation_loop.md`
- relevant memory tooling, retrieval scripts, and benchmark artifacts

## 3) Expertise requested

Best fit:
- retrieval systems engineer,
- infra-minded ML/AI tooling engineer,
- Python integration engineer,
- benchmark/test harness author.

Not primarily asking for:
- theory expansion,
- symbolic interpretation,
- grand conceptual synthesis.

## 4) Required constraints

1. Do not increase the claim tier of topological memory.
2. Do not let retrieval outputs directly alter canonical governance or runtime policy.
3. Preserve inspectable provenance for every retrieval artifact used by the workflow.
4. Keep fallback behavior available if topological retrieval is absent or fails.
5. Prefer one narrow integration path over broad repo-wide entanglement.

## 5) Target gradient

The desired gradient is:

```text
researchy retrieval artifact
  -> stable interface
  -> one bounded workflow consumer
  -> benchmark against baseline
  -> explicit evidence about whether host adoption is justified
```

## 6) Deliverables

Produce the following:

1. **A stable integration interface**
   - a small module, function set, or command entrypoint for topological retrieval
   - inputs and outputs explicitly documented

2. **One bounded workflow consumer**
   - integrate topological retrieval into exactly one repo-native workflow
   - recommended target: planning/orchestration summary generation or continuity-aware task preparation

3. **Baseline comparison path**
   - provide a flat/baseline retrieval mode and a topological mode
   - allow side-by-side comparison on the same small task set

4. **Evidence artifacts**
   - benchmark or comparison outputs
   - example retrieval traces with provenance
   - failure cases where topological retrieval underperforms or degrades clarity

5. **Tests or validation checks**
   - smoke tests for interface behavior
   - at least one regression check for provenance-preserving outputs

6. **A concise implementation note**
   - where the interface lives
   - what workflow now consumes it
   - what did not work
   - what should not be inferred from the results

## 7) Suggested implementation shape

The builder may choose a different design, but the intended shape is something like:

- retrieval interface module
- workflow flag or config toggle
- output artifact written to `memory/` or another inspectable path
- comparison script or test harness
- minimal docs update if the integration path becomes stable

## 8) Acceptance criteria

The packet counts as successful only if all of the following are true:

1. A real Sandy Chaos workflow can call topological retrieval without manual patchwork.
2. That workflow still functions when topological retrieval is disabled or unavailable.
3. Outputs include provenance or inspectable trace structure.
4. There is at least one explicit baseline comparison.
5. The implementation does not let retrieval outputs directly self-authorize planning or governance changes.
6. The result is understandable by someone who did not write it.

## 9) Failure conditions

This packet should be judged a failure, or at least incomplete, if any of the following occur:

- the result is mostly architectural theater with no live consumer,
- integration requires hidden assumptions or operator heroics,
- benchmark outputs are cherry-picked or non-comparable,
- topological retrieval is treated as automatically superior,
- provenance gets weaker rather than stronger,
- or the subsystem gains de facto authority without a declared membrane.

## 10) Non-goals / out-of-scope temptations

Do **not** spend this packet on:

- proving continuity metaphysics,
- redesigning the whole orchestration stack,
- generalizing to every workflow at once,
- replacing all retrieval methods,
- or writing broad philosophical defenses of topological memory.

## 11) Expected artifacts

Examples of acceptable outputs:

- code under `scripts/`, `memory/`, `tools/`, or another justified location
- test or validation artifacts under `tests/`
- benchmark outputs under `memory/` or a clearly named results path
- a small note in `docs/notes/` or `plans/` describing observed outcomes

## 12) Review questions

The reviewer should attack this work with questions like:

1. Is this a real interface or just a demo?
2. What exact workflow now consumes the output?
3. Where is the provenance visible?
4. What happens when retrieval fails or returns junk?
5. Is the baseline comparison honest?
6. Did the implementation accidentally grant authority rather than utility?
7. What evidence would justify promoting this subsystem further, and what evidence says not yet?

## 13) Promotion if successful

If successful, this packet does **not** automatically promote topological memory.

At most, success justifies:
- updating subsystem notes,
- raising workflow participation evidence,
- and considering a next packet for stronger host adoption.

The likely next question would be:

> should topological retrieval remain a bounded optional capability, or is there evidence it deserves default infrastructural status in one lane?
